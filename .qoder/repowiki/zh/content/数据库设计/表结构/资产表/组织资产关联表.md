## 简介
组织资产关联表（organization_main_domains）是本漏洞扫描系统中实现组织与主域名多对多关系的核心桥接表。该表通过外键约束机制，将`organizations`表与`main_domains`表进行灵活关联，支持一个组织管理多个主域名，同时一个主域名也可被多个组织共享。这种设计模式在集团与子公司、多部门协作等复杂业务场景中具有重要价值。

## 核心作用与设计原理
组织资产关联表的核心作用是解耦组织与主域名之间的直接绑定关系，实现灵活的资产管理模式。传统的单向外键设计无法满足多对多的业务需求，而通过引入桥接表，系统获得了更高的数据灵活性和可扩展性。

该表的设计遵循了关系型数据库的第三范式（3NF），避免了数据冗余。通过`organization_id`和`main_domain_id`两个字段的组合主键，确保了每条关联记录的唯一性。同时，`ON DELETE CASCADE`级联删除策略保证了数据的一致性和完整性，当某个组织或主域名被删除时，其所有关联记录也会被自动清除。

## 数据库结构分析
```sql
CREATE TABLE organization_main_domains (
    organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    main_domain_id UUID NOT NULL REFERENCES main_domains(id) ON DELETE CASCADE,
    PRIMARY KEY (organization_id, main_domain_id)
);
```

该表包含两个核心字段：
- **organization_id**: 外键引用`organizations.id`，标识组织实体
- **main_domain_id**: 外键引用`main_domains.id`，标识主域名实体

组合主键的设计确保了同一组织与同一主域名之间只能存在一条关联记录，防止了重复关联。以下示例展示了多个组织与主域名的关联关系：

```sql
-- 组织与主域名关联示例
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES
    ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-000000000008'), -- Example Org 1 -> example1.com
    ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-000000000009'), -- Example Org 1 -> example2.com
    ('00000000-0000-0000-0000-000000000002', '00000000-0000-0000-0000-000000000008'), -- Example Org 2 -> example1.com
    ('00000000-0000-0000-0000-000000000003', '00000000-0000-0000-0000-00000000000a'); -- Tech Solutions Inc. -> xingra.io
```

## Golang ORM映射逻辑
在Golang层面，组织资产关联关系通过`domain.go`文件中的`OrganizationMainDomain`结构体进行映射：

```go
// OrganizationMainDomain 组织主域名关联模型
type OrganizationMainDomain struct {
	OrganizationID string `json:"organization_id" db:"organization_id"`
	MainDomainID   string `json:"main_domain_id" db:"main_domain_id"`
}
```

该结构体使用了GORM风格的标签注解：
- `json:"organization_id"`: 定义JSON序列化时的字段名称
- `db:"organization_id"`: 定义数据库字段映射关系

在`domain-service.go`中，`CreateMainDomains`方法实现了完整的关联逻辑，包括事务处理、存在性检查和批量操作：

```go
// CreateMainDomains 创建主域名并关联到组织
func (s *DomainService) CreateMainDomains(req models.CreateMainDomainsRequest) (*models.APIResponse, error) {
	tx, err := s.db.Begin()
	// ... 事务处理逻辑
	for _, domainName := range req.MainDomains {
		// 检查主域名是否已存在
		var existingID string
		checkQuery := `SELECT id FROM main_domains WHERE main_domain_name = $1`
		err := tx.QueryRow(checkQuery, domainName).Scan(&existingID)
		// ... 创建或获取主域名ID
		// 检查组织是否已关联此主域名
		var associationExists bool
		associationQuery := `
			SELECT EXISTS(
				SELECT 1 FROM organization_main_domains 
				WHERE organization_id = $1 AND main_domain_id = $2
			)
		`
		err = tx.QueryRow(associationQuery, req.OrganizationID, domainID).Scan(&associationExists)
		if !associationExists {
			// 创建组织和主域名的关联
			associateQuery := `
				INSERT INTO organization_main_domains (organization_id, main_domain_id)
				VALUES ($1, $2)
			`
			_, err = tx.Exec(associateQuery, req.OrganizationID, domainID)
		}
	}
	// ... 提交事务
}
```

## 实际关联操作示例
以下是建立组织与主域名关联关系的完整操作流程：

### 1. 创建主域名记录
```sql
INSERT INTO main_domains (id, main_domain_name, created_at)
VALUES ('00000000-0000-0000-0000-00000000000f', 'newcompany.com', NOW());
```

### 2. 建立组织与主域名关联
```sql
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-00000000000f');
```

### 3. 批量关联多个主域名
```sql
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES
    ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-000000000008'),
    ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-000000000009'),
    ('00000000-0000-0000-0000-000000000001', '00000000-0000-0000-0000-00000000000a');
```

### 4. 查询组织关联的所有主域名
```sql
SELECT md.id, md.main_domain_name, md.created_at
FROM main_domains md
INNER JOIN organization_main_domains omd ON md.id = omd.main_domain_id
WHERE omd.organization_id = '00000000-0000-0000-0000-000000000001'
ORDER BY md.created_at DESC;
```

## 查询性能优化策略
为提升组织资产关联表的查询性能，系统在`初始化.sql`脚本中创建了专门的索引：

```sql
-- 为组织ID创建索引
CREATE INDEX IF NOT EXISTS idx_organization_main_domains_org_id ON organization_main_domains(organization_id);

-- 为主域名ID创建索引
CREATE INDEX IF NOT EXISTS idx_organization_main_domains_domain_id ON organization_main_domains(main_domain_id);
```

这些索引显著提升了以下查询场景的性能：
- **按组织查询关联域名**: 通过`organization_id`索引快速定位所有关联记录
- **按域名查询所属组织**: 通过`main_domain_id`索引反向查询组织信息
- **关联查询优化**: 在JOIN操作中，索引可以大幅减少扫描的数据量

对于高并发场景，还可以考虑创建复合索引：
```sql
-- 复合索引适用于同时查询组织和域名的场景
CREATE INDEX idx_organization_main_domains_composite ON organization_main_domains(organization_id, main_domain_id);
```

## 业务场景应用
组织资产关联表在实际业务中支持多种复杂场景：

### 集团与子公司场景
一个集团组织可以关联多个子公司的主域名，实现统一的安全管理：
```sql
-- 集团组织关联多个子公司域名
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES
    ('group-org-id', 'subsidiary1.com-id'),
    ('group-org-id', 'subsidiary2.com-id'),
    ('group-org-id', 'subsidiary3.com-id');
```

### 多部门协作场景
不同部门可以共享同一主域名的扫描结果，避免重复工作：
```sql
-- 多个部门共享同一主域名
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES
    ('security-dept-id', 'company.com-id'),
    ('it-dept-id', 'company.com-id'),
    ('compliance-dept-id', 'company.com-id');
```

### 资产移交场景
当组织结构调整时，可以通过删除和重新关联实现资产的平滑移交：
```sql
-- 移除旧组织的关联
DELETE FROM organization_main_domains 
WHERE organization_id = 'old-org-id' AND main_domain_id = 'domain-id';

-- 建立新组织的关联
INSERT INTO organization_main_domains (organization_id, main_domain_id)
VALUES ('new-org-id', 'domain-id');
```
