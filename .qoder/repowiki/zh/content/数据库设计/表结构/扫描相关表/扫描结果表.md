## 扫描结果表与扫描任务表的关系

`scan_results`表与`scan_tasks`表之间存在明确的一对一关系，由外键约束`scan_task_id`强制实施。每个扫描任务（`scan_task`）在执行完成后，会生成一条且仅一条对应的扫描结果记录。这种设计确保了结果数据的完整性，避免了因多个结果记录关联到同一任务而产生的数据歧义。

从数据库定义可以看出，`scan_results`表中的`scan_task_id`字段被声明为`NOT NULL`并引用`scan_tasks(id)`，同时设置了`ON DELETE CASCADE`级联删除规则。这意味着：
- **数据完整性**：每个结果必须关联到一个存在的扫描任务，不允许孤立的结果记录。
- **级联删除**：当某个扫描任务被删除时，其对应的结果记录也会被自动删除，保持数据一致性。

```sql
CREATE TABLE scan_results (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    scan_task_id UUID NOT NULL REFERENCES scan_tasks(id) ON DELETE CASCADE,
    result_summary TEXT,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL
);
```

这种一对一的关系设计适用于该场景，因为一次扫描任务的执行被视为一个原子操作，其最终结果被汇总为一个摘要。如果系统需要支持单次任务的多次部分扫描或增量扫描，则可能需要调整为一对多关系，但当前设计表明系统采用的是全量扫描和一次性结果生成的模式。

## result_summary字段设计策略

`result_summary`字段被设计为`TEXT`类型，这是一个经过深思熟虑的决策，旨在平衡灵活性、性能和可扩展性。

### TEXT类型的选择原因
选择`TEXT`而非`VARCHAR(N)`的主要原因在于**数据大小的不确定性**。安全扫描的结果摘要可能非常简短（如"扫描成功，未发现漏洞"），也可能非常庞大（如包含数百个子域名的列表和多个漏洞的详细描述）。使用`TEXT`类型可以：
- **避免长度限制**：无需预估最大长度，防止因`VARCHAR`长度不足而导致数据截断或插入失败。
- **存储效率**：在PostgreSQL中，`TEXT`和`VARCHAR`在存储上没有区别，`TEXT`提供了更大的灵活性而没有性能损失。

### JSON格式存储方案
虽然`result_summary`是`TEXT`类型，但其内容很可能采用**JSON格式**进行序列化存储。这种设计策略提供了以下优势：
- **结构化数据**：可以在一个字段中存储复杂的、层次化的结果数据，例如：
  ```json
  {
    "subdomains_found": 125,
    "open_ports": [80, 443, 22],
    "vulnerabilities": [
      {"type": "SQL Injection", "severity": "high", "url": "http://example.com/login"},
      {"type": "XSS", "severity": "medium", "url": "http://example.com/search"}
    ],
    "scan_duration": "15m30s"
  }
  ```
- **查询灵活性**：结合PostgreSQL的`JSONB`类型（虽然此处是`TEXT`，但应用层可解析为JSON），可以在未来升级中支持对结果内容的查询，例如查找所有发现SQL注入漏洞的扫描。
- **前后端交互**：Go的`ScanResult`结构体可以直接将`ResultSummary`字段定义为`string`，在序列化为JSON响应时，前端可以轻松地将此字符串解析为JavaScript对象进行展示。

### 大数据量下的性能考量
当`result_summary`存储大量数据时，需要考虑以下性能影响：
- **I/O开销**：读取包含大文本的结果记录会增加磁盘I/O和网络传输时间。
- **内存占用**：加载大结果摘要会消耗更多应用服务器内存。
- **查询性能**：全表扫描或排序操作会因大字段而变慢。

应对策略包括：
1. **分页查询**：在获取扫描历史时，使用分页（pagination）限制每次返回的记录数。
2. **延迟加载**：在列表视图中只显示摘要的前缀（如"发现 125 个子域名"），点击“查看详情”时才加载完整的`result_summary`。
3. **结果归档**：对于非常庞大的结果，可以考虑将完整报告存储在对象存储（如S3）中，而`result_summary`字段仅存储一个指向该报告的URL。

## created_at字段在审计追踪中的作用

`created_at`字段在`scan_results`表中扮演着至关重要的角色，是实现**结果归档**和**审计追踪**功能的核心。

### 结果归档
`created_at`提供了一个精确的时间戳，用于：
- **历史记录**：准确记录每次扫描完成的时间，形成一个按时间排序的结果历史。
- **趋势分析**：通过比较不同时间点的扫描结果，可以分析资产的变化趋势，例如子域名的增长、新出现的漏洞等。
- **数据保留策略**：可以基于`created_at`字段实现自动归档或清理策略，例如保留最近6个月的结果，更早的数据则移动到冷存储或删除。

### 审计追踪
在安全和合规性要求高的环境中，`created_at`是审计追踪的关键组成部分：
- **事件溯源**：当发现一个安全问题时，可以通过`created_at`追溯到是哪一次扫描首次发现了该问题，以及问题持续了多久。
- **责任认定**：结合`scan_tasks`表中的信息，可以确定是哪个组织、在什么时间发起了扫描，从而明确责任。
- **合规性报告**：满足定期安全扫描的合规要求，证明在特定时间点执行了安全评估。

此外，`updated_at`字段的存在表明结果摘要可能被更新（例如，扫描完成后补充了最终报告），这进一步增强了审计能力，可以追踪结果摘要的任何变更。

## ScanResult结构体与数据库表的映射

`ScanResult`结构体（位于`backend/internal/models/scan.go`）与`scan_results`数据库表之间存在直接的、通过标签（tags）定义的映射关系。这种映射由Go的ORM（如GORM）或数据库驱动库（如`sqlx`）在运行时解析，实现了数据在数据库行和Go对象之间的自动转换。

### 结构体与表的映射关系
```go
// ScanResult 扫描结果模型
type ScanResult struct {
	ID            string    `json:"id" db:"id"`
	ScanTaskID    string    `json:"scan_task_id" db:"scan_task_id"`
	ResultSummary string    `json:"result_summary" db:"result_summary"`
	CreatedAt     time.Time `json:"created_at" db:"created_at"`
	UpdatedAt     time.Time `json:"updated_at" db:"updated_at"`
}
```
- **字段映射**：每个结构体字段通过`db:"column_name"`标签精确地映射到数据库表的列。例如，`ScanTaskID`字段映射到`scan_task_id`列。
- **序列化处理**：`json:"scan_task_id"`标签定义了该结构体在序列化为JSON（例如，通过HTTP API返回给前端）时的字段名。这确保了API的响应格式是`snake_case`（如`scan_task_id`），符合常见的API设计规范，而Go代码内部使用`CamelCase`（如`ScanTaskID`），符合Go的命名约定。

### 序列化处理流程
1. **从数据库读取**：当从数据库查询一行`scan_results`记录时，数据库驱动会根据`db`标签将列的值填充到对应的结构体字段中。
2. **返回给前端**：当此`ScanResult`对象被用作HTTP响应体时，`encoding/json`包会根据`json`标签将结构体序列化为JSON对象。
3. **前端使用**：前端JavaScript代码接收到JSON后，可以直接使用`scan_task_id`等属性进行展示。

这种分离的设计（内部使用`CamelCase`，外部使用`snake_case`）是Go Web开发中的最佳实践，它在保持代码可读性的同时，也保证了API的通用性和兼容性。

## 基于task_id的索引与查询优化

为了优化基于扫描任务的结果检索性能，数据库中为`scan_results`表的`scan_task_id`字段创建了专用索引。

### 索引创建
```sql
CREATE INDEX IF NOT EXISTS idx_scan_results_task_id ON scan_results(scan_task_id);
```
此B-Tree索引极大地加速了以下类型的查询：
- **精确查找**：`SELECT * FROM scan_results WHERE scan_task_id = 'some-uuid';`
- **关联查询**：`SELECT * FROM scan_tasks t JOIN scan_results r ON t.id = r.scan_task_id WHERE t.organization_id = 'org-uuid';`

### 结果数据的分页查询和导出实现
结合索引，系统可以高效地实现分页查询和数据导出：

#### 分页查询
分页查询通常通过`LIMIT`和`OFFSET`或更高效的`cursor-based`（游标）分页来实现。例如，获取某个组织的扫描结果历史：
```sql
SELECT r.*, t.status, t.created_at as task_created_at
FROM scan_results r
JOIN scan_tasks t ON r.scan_task_id = t.id
WHERE t.organization_id = $1
ORDER BY r.created_at DESC
LIMIT $2 OFFSET $3;
```
- **索引利用**：`ORDER BY r.created_at DESC`可以利用`created_at`上的索引（如果存在）来避免排序操作。
- **性能**：`scan_task_id`上的索引确保了`JOIN`操作非常快，而`organization_id`上的索引（`idx_scan_tasks_org_id`）确保了`WHERE`子句的高效过滤。

#### 数据导出
对于导出功能，系统可能会：
1. **执行查询**：使用与分页查询类似的SQL，但可能不带`LIMIT`以获取所有匹配记录。
2. **流式处理**：为了避免将所有结果一次性加载到内存中，可以使用数据库的游标（cursor）进行流式读取。
3. **格式化**：将查询结果逐行格式化为CSV或JSON文件。
4. **提供下载**：将生成的文件通过HTTP响应流式传输给用户。

这种设计确保了即使在处理大量扫描结果时，系统也能保持良好的响应性和内存效率。
