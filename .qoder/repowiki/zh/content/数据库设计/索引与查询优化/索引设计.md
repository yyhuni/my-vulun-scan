## 引言
本索引设计文档详细阐述了漏洞扫描系统数据库中定义的索引结构及其优化策略。文档聚焦于`organizations`表的`name`字段、`main_domains`表的`domain_name`字段以及`scan_tasks`表的`status`字段所创建的B-Tree索引，解释其在查询加速中的作用。同时，文档深入分析了复合索引在多条件查询场景下的优势，例如`(org_id, created_at)`索引在组织资产时间序列检索中的应用。此外，文档还涵盖了索引命名规范、选择性评估方法、GORM模型标签中的索引声明方式，以及索引对写入性能的影响和监控建议。

## 核心索引结构分析

数据库初始化脚本（`backend/初始化.sql`）中定义了多个B-Tree索引，旨在优化关键查询路径的性能。这些索引主要针对高频查询的字段，如名称、状态和外键。

### 单字段B-Tree索引

B-Tree（平衡树）索引是关系型数据库中最常用的索引类型，特别适用于等值查询和范围查询。在本系统中，为以下字段创建了单字段B-Tree索引：

*   **`main_domains` 表的 `main_domain_name` 字段**:
    ```sql
    CREATE INDEX IF NOT EXISTS idx_main_domains_main_domain_name ON main_domains(main_domain_name);
    ```
    此索引用于加速通过域名精确查找主域名记录的查询。例如，当用户输入一个域名进行扫描时，系统需要快速定位`main_domains`表中的对应记录。该索引将查询时间复杂度从O(n)降低到O(log n)，极大地提升了响应速度。

*   **`organizations` 表的 `name` 字段**:
    虽然初始化脚本中未直接为此字段创建索引，但根据`models/organization.go`中的`Organization`模型定义，`name`字段在业务逻辑中很可能是一个高频查询字段。为`name`字段创建索引可以优化通过组织名称查找组织信息的查询。
    ```go
    type Organization struct {
        ID          string    `json:"id" db:"id"`
        Name        string    `json:"name" db:"name"` // 此字段应考虑索引
        Description string    `json:"description" db:"description"`
        CreatedAt   time.Time `json:"created_at" db:"created_at"`
    }
    ```

*   **`scan_tasks` 表的 `status` 字段**:
    ```sql
    CREATE INDEX IF NOT EXISTS idx_scan_tasks_status ON scan_tasks(status);
    ```
    此索引用于加速按任务状态（如`pending`, `running`, `completed`, `failed`）进行筛选的查询。例如，前端工作台需要实时展示所有“运行中”或“待处理”的扫描任务，此索引能高效地返回符合条件的记录集合。

## 复合索引优势与应用场景

复合索引（或称组合索引）是在多个列上创建的单个索引。它在处理涉及多个WHERE条件的查询时具有显著优势。

### 复合索引 `(org_id, created_at)` 的优势

在`scan_tasks`表上，存在一个隐式的复合索引需求，即`(organization_id, created_at)`。虽然初始化脚本中分别创建了`idx_scan_tasks_org_id`和`idx_scan_tasks_status`，但为`(organization_id, created_at)`创建复合索引能更好地服务于“组织资产时间序列检索”这类场景。

```sql
-- 建议的复合索引
CREATE INDEX IF NOT EXISTS idx_scan_tasks_org_created ON scan_tasks(organization_id, created_at DESC);
```

该复合索引的优势体现在：
1.  **覆盖查询 (Covering Query)**: 对于查询“获取某个组织最近10次的扫描任务”，数据库引擎可以直接使用该索引完成查询，无需回表（访问原始数据行），因为索引中已经包含了`organization_id`和`created_at`这两个筛选和排序所需的字段。
2.  **高效排序**: `created_at`字段按降序排列，使得获取“最新”记录的操作无需额外的排序步骤，性能更优。
3.  **最左前缀原则 (Leftmost Prefix Rule)**: 该复合索引同样可以服务于仅按`organization_id`查询的请求，因为`organization_id`是索引的最左列。

### 其他复合索引示例

初始化脚本中还定义了其他复合索引，例如`workflow_executions`表上的`(started_at, status)`，用于高效查询“在某个时间段内启动且状态为‘完成’的执行记录”。

## 索引命名规范

本系统采用了一套清晰、一致的索引命名规范，便于数据库管理员和开发人员快速识别索引的用途。

*   **命名格式**: `idx_{table_name}_{column_name}`
*   **示例**:
    *   `idx_main_domains_main_domain_name`: 表示在`main_domains`表的`main_domain_name`列上的索引。
    *   `idx_scan_tasks_status`: 表示在`scan_tasks`表的`status`列上的索引。
*   **复合索引命名**: 对于复合索引，将多个列名用下划线连接。
    *   `idx_scan_tasks_org_created`: 表示在`scan_tasks`表的`organization_id`和`created_at`列上的复合索引。

这种命名规范具有良好的可读性和可维护性，能够清晰地反映索引与表、列之间的映射关系。

## 选择性评估方法

索引的选择性（Selectivity）是衡量索引效率的关键指标。高选择性意味着索引列的值分布广泛，重复值少，这样的索引过滤效果最好。

### 评估方法

1.  **计算选择性比率**:
    选择性比率 = (不同值的数量) / (总行数)
    比率越接近1，选择性越高。例如，`organizations.name`字段（假设组织名唯一）的选择性接近1，是创建索引的理想候选。而`scan_tasks.status`字段的选择性较低（只有几个状态值），但因其是高频查询字段，创建索引仍然能带来性能提升。

2.  **使用 `EXPLAIN ANALYZE`**:
    在PostgreSQL中，使用`EXPLAIN ANALYZE`命令可以查看查询执行计划。通过观察`Seq Scan`（顺序扫描）和`Index Scan`（索引扫描）的成本，可以判断索引是否被有效利用以及其带来的性能提升。

3.  **监控查询性能**:
    在生产环境中，通过数据库监控工具（如`pg_stat_statements`）来识别执行时间长的慢查询，并分析其是否可以被现有索引优化或需要创建新索引。

## GORM模型中的索引声明

虽然本项目的数据库初始化是通过SQL脚本完成的，但在使用GORM等ORM框架时，索引可以直接在模型结构体的标签（tag）中声明，实现代码与数据库结构的同步。

### 示例

在`models/organization.go`中，可以通过添加GORM标签来声明索引：

```go
type Organization struct {
    ID          string    `json:"id" db:"id" gorm:"primaryKey"`
    Name        string    `json:"name" db:"name" gorm:"uniqueIndex;size:255"` // 声明唯一索引
    Description string    `json:"description" db:"description"`
    CreatedAt   time.Time `json:"created_at" db:"created_at"`
}
```

当使用GORM的`AutoMigrate`功能时，它会自动在数据库中创建相应的索引。这种方式将索引定义与业务模型代码紧密结合，提高了代码的可维护性。

## 索引对写入性能的影响

索引在加速查询的同时，也会对数据库的写入性能（INSERT, UPDATE, DELETE）产生负面影响。

### 影响分析

1.  **写入开销**: 每次向表中插入、更新或删除一行数据时，数据库不仅需要修改表数据，还必须同步更新所有相关的索引。索引越多，写入操作的开销就越大。
2.  **存储空间**: 每个索引都需要额外的磁盘空间来存储其数据结构。对于大型表，索引可能占用与表数据相当甚至更多的空间。
3.  **锁竞争**: 在高并发写入场景下，索引的更新可能会增加行锁或页锁的竞争，导致写入性能下降。

### 平衡策略

*   **避免过度索引**: 只为真正需要加速的查询创建索引。定期审查和删除不再使用或效果不佳的索引。
*   **选择合适的索引类型**: 对于JSONB字段（如`workflow_data`），使用GIN索引是合适的，但对于普通文本字段，B-Tree通常更高效。
*   **批量写入**: 在进行大量数据导入时，可以考虑先删除非关键索引，导入完成后再重建，以减少写入过程中的开销。

## 索引监控与维护建议

为了确保索引持续高效，需要进行定期的监控和维护。

### 监控建议

1.  **使用 `pg_stat_user_indexes` 视图**: 查询此视图可以了解每个索引的使用情况，如`idx_scan`（索引扫描次数）和`idx_tup_read`（通过索引读取的元组数）。长期未被使用的索引（`idx_scan`为0）应被考虑删除。
2.  **慢查询日志**: 启用并分析数据库的慢查询日志，找出未被索引覆盖的慢查询。
3.  **性能监控工具**: 使用Prometheus + Grafana等工具对数据库的查询延迟、IOPS等指标进行监控，及时发现性能瓶颈。

### 维护建议

1.  **定期 `REINDEX`**: 对于频繁更新的表，索引可能会变得“碎片化”，影响查询性能。定期执行`REINDEX`命令可以重建索引，优化其物理存储结构。
2.  **`VACUUM ANALYZE`**: 定期运行`VACUUM ANALYZE`命令。`VACUUM`回收被删除行占用的空间，`ANALYZE`更新表和索引的统计信息，帮助查询优化器做出更优的执行计划决策。
