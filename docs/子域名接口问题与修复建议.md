## 发现的问题

### 1. 【重要】排序字段映射不完整

**位置**: `backend/internal/services/subdomain.go`

**涉及方法**:
- `GetSubDomains()` (第 28-85 行)
- `GetSubDomainsByDomainID()` (第 87-145 行)
- `GetSubDomainsByOrgID()` (第 147-209 行)

**问题描述**:

三个方法中的 `sortFieldMap` 只映射了驼峰命名（`createdAt`/`updatedAt`），未映射下划线命名（`created_at`/`updated_at`）。

```go
// 当前映射（不完整）
sortFieldMap := map[string]string{
    "id":        "id",
    "name":      "name",
    "createdAt": "created_at",
    "updatedAt": "updated_at",
}
```

**影响**:

当前端传递 `sort_by=created_at` 或 `sort_by=updated_at` 时（下划线格式），映射表找不到对应键，会 fallback 到默认的 `updated_at`，导致排序参数不生效。

**修复方案**:

补全下划线键的映射：

```go
sortFieldMap := map[string]string{
    "id":         "id",
    "name":       "name",
    "createdAt":  "created_at",
    "updatedAt":  "updated_at",
    "created_at": "created_at",  // 新增
    "updated_at": "updated_at",  // 新增
}
```

**修复文件**:
- `backend/internal/services/subdomain.go` 的三个方法中的 `sortFieldMap`

### 2. 【重要】唯一索引定义有冲突风险

**位置**: `backend/internal/models/subdomain.go` (第 17-18 行)

**问题描述**:

`SubDomain` 模型的 `Name` 字段同时定义了两种唯一索引：

```go
Name     string `json:"name" gorm:"not null;size:255;uniqueIndex;uniqueIndex:idx_subdomain_domain_name;..."`
DomainID uint   `json:"domain_id" gorm:"not null;index;uniqueIndex:idx_subdomain_domain_name;..."`
```

这会创建：
1. **单列唯一索引**: `Name` 字段全局唯一
2. **联合唯一索引**: `(DomainID, Name)` 组合唯一

**影响**:

单列唯一索引会导致不同域名下无法有相同的子域名。例如：
- `a.example.com` (domain_id=1)
- `a.other.com` (domain_id=2)

第二条记录会因为 `Name='a.other.com'` 与第一条冲突而插入失败（如果两者的 `Name` 相同）。

**业务预期**:

通常我们只需要"在同一个 Domain 下子域名唯一"，即联合唯一索引即可。

**修复方案**:

去掉 `Name` 字段上的单列 `uniqueIndex`，只保留联合唯一索引：

```go
// 修复后
Name     string `json:"name" gorm:"not null;size:255;uniqueIndex:idx_subdomain_domain_name;check:name = LOWER(name);<-:create"`
DomainID uint   `json:"domain_id" gorm:"not null;index;uniqueIndex:idx_subdomain_domain_name;<-:create"`
```

**修复文件**:
- `backend/internal/models/subdomain.go` 第 17 行

### 3. 【可选优化】组织子域名接口的 404 语义

**位置**: `backend/internal/handlers/subdomain.go` (第 183 行)

**问题描述**:

`GetSubDomainsByOrgID()` 的 Swagger 注释声明可能返回 404 "组织不存在"：

```go
// @Failure 404 {object} models.APIResponse "组织不存在"
```

但 Handler 实现中未显式检查组织是否存在，直接通过 JOIN 查询子域名列表。如果组织不存在，会返回空列表（200 状态码），而不是 404。

**影响**:

注释与实际行为不一致，可能误导 API 使用者。

**修复方案（二选一）**:

**方案 A**: 调整注释以匹配现状（推荐，符合项目"不做安全校验"的规则）

```go
// @Failure 500 {object} models.APIResponse "服务器内部错误"
// 移除 404 的注释
```

**方案 B**: 在 Handler 层前置校验组织存在性

```go
// 在调用 service 前添加
orgService := services.NewOrganizationService()
if _, err := orgService.GetOrganizationByID(uri.ID); err != nil {
    response.NotFoundResponse(c, "组织不存在")
    return
}
```

**建议**: 采用方案 A，保持简洁，符合项目规则。

### 4. 【可选优化】删除接口错误码语义

**位置**: `backend/internal/handlers/subdomain.go` (第 286-310 行)

**问题描述**:

`BatchDeleteSubDomains()` 在"部分子域名 ID 不存在"的情况下，Service 层返回错误，Handler 统一用 500 返回：

```go
if err != nil {
    response.InternalServerErrorResponse(c, "批量删除子域名失败: "+err.Error())
    return
}
```

但 Service 层的错误是业务逻辑错误（"部分子域名ID不存在"），更符合 400（Bad Request）或 404（Not Found）语义。

**影响**:

500 通常表示服务器内部错误，用于数据库连接失败等场景。业务校验失败使用 500 会混淆错误类型。

**修复方案**:

在 Handler 层区分错误类型：

```go
subdomains, err := service.BatchDeleteSubDomains(req.SubDomainIDs)
if err != nil {
    // 如果错误信息包含"不存在"，返回 400
    if strings.Contains(err.Error(), "不存在") {
        response.BadRequestResponse(c, err.Error())
        return
    }
    // 其他错误返回 500
    response.InternalServerErrorResponse(c, "批量删除子域名失败: "+err.Error())
    return
}
```

或在 Service 层返回自定义错误类型，Handler 层根据错误类型选择响应码。

### 5. 【可选优化】创建接口返回体字段补全

**位置**: `backend/internal/handlers/subdomain.go` (第 360-363 行)

**问题描述**:

`CreateSubDomainsForDomain()` 返回 `models.CreateSubDomainsResponseData`，该结构体包含以下字段：

```go
type CreateSubDomainsResponseData struct {
    SubdomainsCreated     int      `json:"subdomains_created"`      // 实际创建的子域名数量
    AlreadyExists         int      `json:"already_exists"`          // 已存在的子域名数量
    SkippedDomains        []string `json:"skipped_domains"`         // 被跳过的根域名列表
    TotalUniqueSubdomains int      `json:"total_unique_subdomains"` // 请求的唯一子域名总数
}
```

当前 Handler 只填充了 `SubdomainsCreated` 和 `AlreadyExists`：

```go
response.SuccessResponse(c, models.CreateSubDomainsResponseData{
    SubdomainsCreated: result.SubdomainsCreated,
    AlreadyExists:     alreadyExists,
})
```

**影响**:

`SkippedDomains` 和 `TotalUniqueSubdomains` 字段为零值，前端无法获取完整信息。

**修复方案**:

补全所有字段：

```go
response.SuccessResponse(c, models.CreateSubDomainsResponseData{
    SubdomainsCreated:     result.SubdomainsCreated,
    AlreadyExists:         alreadyExists,
    SkippedDomains:        []string{},  // 当前场景无跳过的域名
    TotalUniqueSubdomains: result.TotalUniqueSubdomains,  // Service 层已有此字段
})
```

## Curl 测试命令（端口 8888）

### 1. 获取所有子域名（分页/排序）
```bash
curl -s "http://localhost:8888/api/v1/subdomains?page=1&page_size=10&sort_by=updated_at&sort_order=desc"
```

### 2. 获取子域名详情
```bash
curl -s "http://localhost:8888/api/v1/subdomains/123"
```

### 3. 按域名获取子域名
```bash
curl -s "http://localhost:8888/api/v1/domains/1/subdomains?page=1&page_size=10&sort_by=created_at&sort_order=asc"
```

### 4. 按组织获取子域名
```bash
curl -s "http://localhost:8888/api/v1/organizations/2/subdomains?page=1&page_size=20&sort_by=name&sort_order=asc"
```

### 5. 删除单个子域名
```bash
curl -s -X DELETE "http://localhost:8888/api/v1/subdomains/123"
```

### 6. 批量删除子域名
```bash
curl -s -X POST "http://localhost:8888/api/v1/subdomains/batch-delete" \
  -H "Content-Type: application/json" \
  -d '{"subdomain_ids":[1,2,3]}'
```

### 7. 为域名批量创建子域名
```bash
curl -s -X POST "http://localhost:8888/api/v1/domains/1/subdomains/create" \
  -H "Content-Type: application/json" \
  -d '{"subdomains":["a.example.com","b.example.com","a.example.com"]}'
```
