### ⚠️ 3.1 数据库连接池配置不合理

**严重程度：** ⭐⭐⭐  
**位置：** `backend/pkg/database/database.go:105-107`

#### 问题描述

当前连接池配置可能导致性能问题和资源浪费。

#### 问题代码

```go
// 配置连接池
sqlDB.SetMaxOpenConns(cfg.Database.MaxConns)          // 假设为 10
sqlDB.SetMaxIdleConns(cfg.Database.MaxConns / 2)      // 5 个空闲连接
sqlDB.SetConnMaxLifetime(5 * time.Minute)             // 5 分钟后强制关闭
```

#### 问题分析

1. **空闲连接数过少**：`MaxIdleConns = MaxConns / 2` 导致：
   - 频繁创建和销毁连接（性能开销）
   - 突发流量时响应变慢
   
2. **连接生命周期过短**：5 分钟可能导致：
   - 频繁重建连接
   - 在高负载时增加数据库压力

3. **缺少空闲超时**：没有设置 `ConnMaxIdleTime`，空闲连接永不释放

#### 影响范围

- **性能**：高并发时频繁创建连接，增加延迟
- **资源**：可能出现连接泄漏

#### 优化方案

```go
// 配置连接池 - 优化版
func configureConnectionPool(sqlDB *sql.DB, cfg *config.Config) {
    // 最大打开连接数
    maxOpen := cfg.Database.MaxConns
    if maxOpen == 0 {
        maxOpen = 25  // 默认值
    }
    sqlDB.SetMaxOpenConns(maxOpen)
    
    // ✅ 最大空闲连接数应该接近最大连接数
    // 避免频繁创建和销毁连接
    sqlDB.SetMaxIdleConns(maxOpen)
    
    // ✅ 连接最大空闲时间（10 分钟未使用则关闭）
    // 防止长期空闲连接占用资源
    sqlDB.SetConnMaxIdleTime(10 * time.Minute)
    
    // ✅ 连接最大生命周期（1 小时后强制关闭重建）
    // 防止连接长期持有导致的问题
    sqlDB.SetConnMaxLifetime(1 * time.Hour)
    
    log.Info().
        Int("max_open", maxOpen).
        Int("max_idle", maxOpen).
        Dur("max_idle_time", 10*time.Minute).
        Dur("max_lifetime", 1*time.Hour).
        Msg("Database connection pool configured")
}

// 在 InitDB 中调用
sqlDB, err = db.DB()
if err != nil {
    // ...
}

configureConnectionPool(sqlDB, cfg)
```

#### 推荐配置

| 场景 | MaxOpenConns | MaxIdleConns | ConnMaxIdleTime | ConnMaxLifetime |
|------|--------------|--------------|-----------------|-----------------|
| 开发环境 | 10 | 10 | 10 分钟 | 1 小时 |
| 测试环境 | 25 | 25 | 10 分钟 | 1 小时 |
| 生产环境（小） | 50 | 50 | 15 分钟 | 2 小时 |
| 生产环境（大） | 100 | 100 | 15 分钟 | 2 小时 |

### ⚠️ 3.2 错误处理使用字符串匹配

**严重程度：** ⭐⭐⭐  
**位置：** `backend/internal/handlers/domain.go:239-252`

#### 问题描述

通过字符串匹配错误类型不可靠且难以维护。

#### 问题代码

```go
err := service.DeleteDomainFromOrganization(req)
if err != nil {
    // ❌ 字符串匹配错误
    switch err.Error() {
    case "organization not found":
        response.NotFoundResponse(c, fmt.Sprintf("组织 ID: %d 不存在", uri.OrganizationID))
        return
    case "domain not found":
        response.NotFoundResponse(c, fmt.Sprintf("域名 ID: %d 不存在", uri.DomainID))
        return
    case "association not found":
        response.NotFoundResponse(c, fmt.Sprintf("组织 ID: %d 与域名 ID: %d 之间不存在关联关系", uri.OrganizationID, uri.DomainID))
        return
    default:
        response.InternalServerErrorResponse(c, fmt.Sprintf("从组织 ID: %d 移除域名 ID: %d 失败: %s", uri.OrganizationID, uri.DomainID, err.Error()))
        return
    }
}
```

#### 问题分析

1. **脆弱性**：如果 Service 层修改错误消息，Handler 会失效
2. **不符合 Go 最佳实践**：应该使用 `errors.Is` 或 `errors.As`
3. **国际化困难**：错误消息硬编码在多处

#### 优化方案

```go
// 1. 在 internal/errors/errors.go 中定义标准错误
package errors

import (
    "errors"
    "fmt"
)

var (
    ErrOrganizationNotFound = errors.New("organization not found")
    ErrDomainNotFound       = errors.New("domain not found")
    ErrSubdomainNotFound    = errors.New("subdomain not found")
    ErrEndpointNotFound     = errors.New("endpoint not found")
    ErrAssociationNotFound  = errors.New("association not found")
    
    // 可以附加上下文的错误
    ErrDuplicateEntry = errors.New("duplicate entry")
    ErrInvalidInput   = errors.New("invalid input")
)

// 带上下文的错误包装
func WrapNotFound(entityType string, id uint) error {
    var baseErr error
    switch entityType {
    case "organization":
        baseErr = ErrOrganizationNotFound
    case "domain":
        baseErr = ErrDomainNotFound
    default:
        baseErr = errors.New("not found")
    }
    return fmt.Errorf("%w: %s id=%d", baseErr, entityType, id)
}
```

**严重程度：** ⭐⭐⭐  
**位置：** `backend/internal/handlers/endpoint.go`

#### 问题描述

Endpoint 创建时没有验证 URL 格式，可能导致无效数据进入数据库。

#### 当前代码

```go

    result, err := service.CreateEndpoints(req)
    // ...
}
```

#### 优化方案

```go
// 1. 在 internal/utils/url_validator.go 中添加验证函数
package utils

import (
    "fmt"
    "net/url"
    "strings"
)

// ValidateURL 验证 URL 格式
func ValidateURL(rawURL string) error {
    if rawURL == "" {
        return fmt.Errorf("URL 不能为空")
    }
    
    // 解析 URL
    parsedURL, err := url.Parse(rawURL)
    if err != nil {
        return fmt.Errorf("无效的 URL 格式: %w", err)
    }
    
    // 检查协议
    if parsedURL.Scheme == "" {
        return fmt.Errorf("URL 必须包含协议 (http/https)")
    }
    if parsedURL.Scheme != "http" && parsedURL.Scheme != "https" {
        return fmt.Errorf("仅支持 http 和 https 协议")
    }
    
    // 检查主机名
    if parsedURL.Host == "" {
        return fmt.Errorf("URL 必须包含主机名")
    }
    
    // 检查长度
    if len(rawURL) > 2048 {
        return fmt.Errorf("URL 长度超过限制 (最大 2048 字符)")
    }
    
    return nil
}

// ValidateHTTPMethod 验证 HTTP 方法
func ValidateHTTPMethod(method string) error {
    validMethods := map[string]bool{
        "GET": true, "POST": true, "PUT": true, "DELETE": true,
        "PATCH": true, "HEAD": true, "OPTIONS": true,
    }
    
    method = strings.ToUpper(method)
    if !validMethods[method] {
        return fmt.Errorf("无效的 HTTP 方法: %s", method)
    }
    
    return nil
}
```

```go
// 2. 在 Handler 中使用验证

            return
        }
    }

    // ...
```

### ⚠️ 3.4 前端缺少请求取消机制

**严重程度：** ⭐⭐  
**位置：** `front/lib/api-client.ts`

#### 问题描述

长时间运行的请求没有取消机制，组件卸载后请求仍在进行，可能导致内存泄漏和无效的状态更新。

#### 问题场景

```typescript
// 用户快速切换页面时
function MyComponent() {
  useEffect(() => {
    api.get('/slow-endpoint').then(data => {
      // ❌ 组件已卸载，但回调仍然执行
      setState(data)
    })
  }, [])
  
  // 没有清理逻辑
}
```

#### 优化方案

```typescript
// 1. 扩展 api-client.ts
import axios, { CancelTokenSource } from 'axios'

// 为每个请求创建可取消的版本
interface CancellablePromise<T> extends Promise<T> {
  cancel: () => void
}

function makeCancellable<T>(
  requestFn: (config: any) => Promise<T>,
  config: any = {}
): CancellablePromise<T> {
  const controller = new AbortController()
  
  const promise = requestFn({
    ...config,
    signal: controller.signal
  }) as CancellablePromise<T>
  
  // 添加 cancel 方法
  promise.cancel = () => {
    controller.abort()
  }
  
  return promise
}

export const api = {
  get: <T = any>(url: string, config?: any): CancellablePromise<T> => {
    return makeCancellable(
      (cfg) => apiClient.get<T>(url, cfg),
      config
    )
  },
  
  post: <T = any>(url: string, data?: any, config?: any): CancellablePromise<T> => {
    return makeCancellable(
      (cfg) => apiClient.post<T>(url, data, cfg),
      config
    )
  },
  
  // ... 其他方法类似
}
```

### ⚠️ 3.5 数据库事务缺少超时控制

**严重程度：** ⭐⭐  
**位置：** 所有使用事务的 Service 方法

#### 问题描述

长时间运行的事务可能导致数据库锁定，影响其他操作。

#### 问题代码

```go
// 当前所有事务都没有超时控制
err := s.db.Transaction(func(tx *gorm.DB) error {
    // 可能执行很长时间...
    return nil
})
```

#### 优化方案

```go
// 1. 在 pkg/database 中添加带超时的事务辅助函数
package database

import (
    "context"
    "time"
    "gorm.io/gorm"
)

// WithTimeout 执行带超时的事务
func WithTimeout(db *gorm.DB, timeout time.Duration, fn func(*gorm.DB) error) error {
    ctx, cancel := context.WithTimeout(context.Background(), timeout)
    defer cancel()
    
    return db.WithContext(ctx).Transaction(fn)
}
```

