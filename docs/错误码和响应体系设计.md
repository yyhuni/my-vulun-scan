# 统一的错误码和响应体系设计方案

## 📌 问题背景

### 当前存在的问题

1. **后端错误消息不友好**
   ```go
   // ❌ 直接暴露技术细节
   response.ValidationErrorResponse(c, "请求参数错误: binding element must be a pointer")
   response.InternalServerErrorResponse(c, "获取组织详情失败: record not found")
   ```
   - 用户看到晦涩的技术术语
   - 暴露了数据库、框架等内部实现细节

2. **消息格式不统一**
   ```go
   "请求参数错误: " + err.Error()  // 有冒号，有前缀
   "组织不存在"                     // 纯文本
   "获取失败: " + err.Error()       // 格式又不同
   ```

3. **前端提示参差不齐**
   - 完全依赖后端返回的消息质量
   - 同样的错误，不同接口提示可能不一致

4. **缺少错误分类**
   - 只有 HTTP 状态码，没有业务错误码
   - 难以区分具体的业务错误类型

---

## 🎯 设计目标

1. **用户友好**：所有错误都用简洁、易懂的中文提示
2. **技术细节隔离**：技术错误信息仅在开发环境日志中记录
3. **统一规范**：前后端错误消息完全一致
4. **易于维护**：集中管理错误码和消息
5. **国际化准备**：使用错误码作为 key，方便扩展多语言

## 🏗️ 系统架构

### 1. 后端错误码体系

#### 错误码规则

```
格式：EXXXX（E + 4位数字）
分类：
- E1xxx：组织模块
- E2xxx：域名模块
- E3xxx：子域名模块
- E4xxx：端点模块
- E5xxx：漏洞模块（预留）
- E9xxx：通用错误
```

#### 文件结构

```
backend/internal/errors/errors.go
├── 错误码常量定义
├── 错误消息映射表
└── 业务错误类型（向后兼容）
```

#### 代码示例

```go
// 错误码常量
const (
    // 通用错误 (E9xxx)
    CodeBadRequest          = "E9001"  // 请求参数有误
    CodeValidationError     = "E9002"  // 数据验证失败
    CodeNotFound            = "E9003"  // 资源不存在
    CodeInternalServerError = "E9004"  // 服务器内部错误

    // 组织模块 (E1xxx)
    CodeOrgNotFound      = "E1001"  // 组织不存在
    CodeOrgCreateFailed  = "E1002"  // 创建组织失败
    CodeOrgUpdateFailed  = "E1003"  // 更新组织失败
    // ... 更多
)

// 消息映射表
var ErrorMessages = map[string]string{
    CodeOrgNotFound:     "组织不存在",
    CodeOrgCreateFailed: "创建组织失败",
    // ... 更多
}

// 获取用户友好消息
func GetErrorMessage(code string) string {
    if msg, ok := ErrorMessages[code]; ok {
        return msg
    }
    return ErrorMessages[CodeInternalServerError]
}
```

```typescript
export const getErrorMessage = (error: any): string => {
  // 1. 请求取消
  if (axios.isCancel(error)) {
    return '请求已被取消'
  }
  
  // 2. 请求超时
  if (error.code === 'ECONNABORTED') {
    return '请求超时，请稍后重试'
  }
  
  // 3. 后端错误码（优先级最高）
  if (error.response?.data?.code) {
    const errorCode = error.response.data.code
    if (typeof errorCode === 'string' && errorCode.startsWith('E')) {
      const mappedMessage = getMessageByErrorCode(errorCode)
      if (mappedMessage) {
        return mappedMessage  // ✅ 返回映射的用户友好消息
      }
    }
  }
  
  // 4. 后端消息
  if (error.response?.data?.message) {
    return error.response.data.message
  }
  
  // 5. HTTP 状态码友好提示
  // 6. 网络错误
  // 7. 兜底消息
  return '发生未知错误'
}
```

### 后端 Handler 写法

#### 示例 1：参数验证错误

```go
func GetOrgByID(c *gin.Context) {

    // 参数绑定失败
    if err := c.ShouldBindUri(&uri); err != nil {
        // ✅ 使用错误码 + 技术详情

        return

    // ... 业务逻辑
}
```

**返回的 JSON**：
```json

  "code": "E9002",
  "state": "error",
  "message": "数据验证失败"
}
```

**开发环境日志**：

#### 示例 2：资源不存在

```go
func GetOrgByID(c *gin.Context) {
    // ... 参数验证
    
    organization, err := service.GetOrgByID(uri.ID)
    if err != nil {
        // 判断是否为"资源不存在"错误
        if errors.Is(err, customErrors.ErrOrganizationNotFound) {
            // ✅ 资源不存在，不需要技术详情
            response.NotFoundResponse(c, customErrors.CodeOrgNotFound)
            return
        }
        
        // ✅ 其他内部错误，传入技术详情用于排查
        response.InternalServerErrorResponse(c, customErrors.CodeOrgListFailed, err.Error())
        return
    }
    
    // 成功
    response.SuccessResponse(c, organization)
}
```

**资源不存在时返回**：
```json

  "code": "E1001",
  "state": "error",
  "message": "组织不存在"
}
```

**内部错误时返回**：
```json
{
  "code": "E1005",
  "state": "error",
  "message": "获取组织列表失败"
}
```

#### 示例 3：业务验证错误

```go
func CreateDomain(c *gin.Context) {
    var req models.CreateDomainsRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.ValidationErrorResponse(c, customErrors.CodeValidationError, err.Error())
        return
    }
    
    // 业务验证
    if len(req.Domains) == 0 {
        // ✅ 业务规则错误
        response.BadRequestResponse(c, customErrors.CodeBadRequest, "域名列表为空")
        return
    }
    
    // 格式验证
    if err := utils.ValidateDomain(req.Domains[0].Name); err != nil {
        // ✅ 格式验证错误，使用专门的错误码
        response.ValidationErrorResponse(c, customErrors.CodeDomainValidationFailed, err.Error())
        return
    }
    
    // ... 创建逻辑
}
```

### 前端使用（无需修改现有代码）

```typescript
// Hook 中的错误处理
export function useCreateOrganization() {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: (data) => OrganizationService.createOrganization(data),
    onMutate: () => {
      toast.loading('正在创建组织...', { id: 'create-org' })
    },
    onSuccess: (response) => {
      toast.dismiss('create-org')
      if (response.state === 'success') {
        toast.success('创建成功')
        queryClient.invalidateQueries({ queryKey: ['organizations'] })
      }
    },
    onError: (error: any) => {
      toast.dismiss('create-org')
      
      // ✅ getErrorMessage 自动处理错误码
      // 如果后端返回 E1002，会自动显示"创建组织失败"
      const message = getErrorMessage(error)
      toast.error(message)
    },
  })
}
```

**用户看到的提示**：
- 成功：`创建成功` ✅
- 失败：`创建组织失败` ❌（而不是 `创建组织失败: database connection timeout`）

## 🔄 迁移指南

### 第一步：识别错误类型

在迁移之前，先理解错误的类型：

1. **参数验证错误** → `CodeValidationError` (E9002)
2. **资源不存在** → 具体模块的 NotFound 码（如 E1001）
3. **创建失败** → 具体模块的 CreateFailed 码（如 E1002）
4. **更新失败** → 具体模块的 UpdateFailed 码（如 E1003）
5. **删除失败** → 具体模块的 DeleteFailed 码（如 E1004）
6. **列表查询失败** → 具体模块的 ListFailed 码（如 E1005）
7. **格式验证失败** → 具体模块的 ValidationFailed 码（如 E2006）

### 第二步：替换错误响应

#### 模式 1：参数绑定错误

```go
// ❌ 旧写法
if err := c.ShouldBindJSON(&req); err != nil {
    response.ValidationErrorResponse(c, "请求参数错误: "+err.Error())
    return
}

// ✅ 新写法
if err := c.ShouldBindJSON(&req); err != nil {
    response.ValidationErrorResponse(c, customErrors.CodeValidationError, err.Error())
    return
}
```

#### 模式 2：资源不存在

```go
// ❌ 旧写法
if errors.Is(err, customErrors.ErrDomainNotFound) {
    response.NotFoundResponse(c, "域名不存在")
    return
}

// ✅ 新写法
if errors.Is(err, customErrors.ErrDomainNotFound) {
    response.NotFoundResponse(c, customErrors.CodeDomainNotFound)
    return
}
```

#### 模式 3：操作失败

```go
// ❌ 旧写法
if err != nil {
    response.InternalServerErrorResponse(c, "创建域名失败: "+err.Error())
    return
}

// ✅ 新写法
if err != nil {
    response.InternalServerErrorResponse(c, customErrors.CodeDomainCreateFailed, err.Error())
    return
}
```

#### 模式 4：格式验证失败

```go
// ❌ 旧写法
if err := utils.ValidateDomain(name); err != nil {
    response.ValidationErrorResponse(c, "域名格式验证失败: "+err.Error())
    return
}

// ✅ 新写法
if err := utils.ValidateDomain(name); err != nil {
    response.ValidationErrorResponse(c, customErrors.CodeDomainValidationFailed, err.Error())
    return
}
```

使用编辑器的查找替换功能：

```go
func GetErrorMessage(code string, lang string) string {

    if msg, ok := messages[code]; ok {

    }
    return messages[CodeInternalServerError]
```

## 📌 注意事项

### 1. 技术详情的使用原则

```go
// ✅ 好的做法：参数验证失败，传入原始错误
if err := c.ShouldBindJSON(&req); err != nil {
    response.ValidationErrorResponse(c, customErrors.CodeValidationError, err.Error())
    return
}

// ✅ 好的做法：资源不存在，不需要技术详情
if errors.Is(err, customErrors.ErrOrgNotFound) {
    response.NotFoundResponse(c, customErrors.CodeOrgNotFound)
    return
}

// ❌ 不好的做法：暴露技术细节给用户
response.NotFoundResponse(c, customErrors.CodeOrgNotFound, "record not found in organizations table")
```

### 2. 向后兼容

- 旧代码可以继续使用 `Legacy*` 函数
- 新功能必须使用新的错误码体系
- 逐步迁移现有代码

### 3. 前端无需修改

- 所有现有的 Hook 代码无需修改
- `getErrorMessage` 函数自动处理新旧格式
- 向后兼容旧版 API 响应

## 🎉 总结

这套错误码体系的核心优势：

1. **用户友好** - 所有错误都是简洁的中文提示
2. **开发友好** - 统一的错误码，易于维护
3. **向后兼容** - 旧代码继续工作，新代码使用新体系
4. **国际化就绪** - 使用错误码作为 key，轻松扩展多语言
5. **前端无感知** - 前端代码无需修改，自动处理新错误码

通过这套体系，您的应用将拥有专业、一致的错误提示体验！
