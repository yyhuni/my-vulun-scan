# 后端排序与分页 - 性能评估与优化方案

## 现状定位（代码中的“自动排序”）
- **Endpoints**
  - `backend/internal/handlers/endpoint.go:setEndpointDefaults()`：默认 `sort_by=updated_at`、`sort_order=desc`。
  - `backend/internal/services/endpoint.go:GetEndpoints()` 与 `GetEndpointsBySubdomainID()`：使用 `buildOrderClause()`；空/非法字段回退 `updated_at desc`。
- **SubDomains**
  - `backend/internal/handlers/subdomain.go:GetSubDomains*`：默认 `updated_at desc`。
  - `backend/internal/services/subdomain.go:GetSubDomains*`：白名单映射，未知字段回退 `updated_at`。
  - 特例：`GetSubDomainsByDomain()` 固定 `Order("created_at DESC")`，且无分页。
- **Domains**
  - `backend/internal/services/domain.go:GetDomainsByOrgID()`、`GetAllDomains()`：`buildOrderClause()`，非法字段回退 `domains.updated_at desc`。
- **Organizations**
  - `backend/internal/services/organization.go:GetOrgs()`：`buildOrderClause()`，非法字段回退 `updated_at desc`。

结论：多数列表接口未显式指定时会“自动”按 `updated_at desc` 排序；`GetSubDomainsByDomain()` 存在“固定排序 + 无分页”。

## 性能影响评估
- **无索引排序**：`updated_at desc` 在大表上易触发外部排序；OFFSET 越大越慢。
- **深页问题**：传统 `LIMIT/OFFSET` 深翻页需扫描/丢弃大量记录，尾延时高。
- **写入成本**：新增 `updated_at` 索引会增加更新开销（可接受，需权衡）。
- **顺序稳定性**：仅按 `updated_at` 可能抖动，建议加次序键 `id`。
- **风险点**：`GetSubDomainsByDomain()` 无分页在数据量大时将显著放大开销。

## 建议方案（按需采纳）
- **A. 建索引（低入侵，见效快）**
  - Endpoints：
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_endpoints_updated_at ON endpoints (updated_at DESC, id DESC);`
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_endpoints_subdomain_updated_at ON endpoints (subdomain_id, updated_at DESC, id DESC);`
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_endpoints_domain_updated_at ON endpoints (domain_id, updated_at DESC, id DESC);`
  - SubDomains：
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sub_domains_updated_at ON sub_domains (updated_at DESC, id DESC);`
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_sub_domains_domain_updated_at ON sub_domains (domain_id, updated_at DESC, id DESC);`
  - Domains：
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_domains_updated_at ON domains (updated_at DESC, id DESC);`
  - Organizations：
    - `CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_organizations_updated_at ON organizations (updated_at DESC, id DESC);`
- **B. 稳定排序（小入侵）**
  - 统一改为：`ORDER BY updated_at DESC, id DESC` 或 `ORDER BY created_at DESC, id DESC`。
  - 修正 `GetSubDomainsByDomain()`：补齐分页与排序参数，默认与其他接口一致（或保留 `created_at` 但加 `id` 次序键）。
- **C. Keyset（游标）分页（中入侵）**
  - 为深翻页接口（如 `/endpoints`）新增可选游标：`(last_updated_at, last_id)`。
  - 查询示例：
    ```sql
    SELECT * FROM endpoints
    WHERE (updated_at, id) < ($1, $2)
    ORDER BY updated_at DESC, id DESC
    LIMIT $pageSize;
    ```
  - 前端保留 offset 兼容，深翻页/滚动加载时启用游标。
- **D. 默认策略取舍**
  - 若“最近更新优先”非刚需，可考虑默认 `id desc` 或 `created_at desc`（更稳定）。若保留 `updated_at desc`，建议同步落地 A+B。

## 验证建议
- 使用 `EXPLAIN (ANALYZE, BUFFERS)` 对比索引前后与深页/过滤场景。
- 重点观察：排序方法（索引扫描 vs 外部排序）、排序内存、共享块命中率、总耗时。

## 最小变更优先级（推荐执行顺序）
1) 先落地 Endpoints 与 SubDomains 的关键索引（A）。
2) 统一排序子句增加 `id` 次序键（B）。
3) 为 `GetSubDomainsByDomain()` 增加分页与排序参数（B）。
4) 视深页需求再评估启用 Keyset 分页（C）。
