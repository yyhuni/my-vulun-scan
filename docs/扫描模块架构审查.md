# 扫描模块架构审查报告

> 审查范围：`apps/scan` 目录下的 signals、tasks、services、orchestrators 模块  
> 审查日期：2024-11-04  
> 审查结论：整体架构清晰，职责分离明确，但存在 9 个需要优化的问题

## 架构设计优点

### ✅ 职责分离清晰

| 模块 | 职责 | 不负责 |
|------|------|--------|
| **Orchestrator** | 编排工作流（匹配、构建） | 执行、状态更新 |
| **Tasks** | 执行业务逻辑 | 状态更新、清理、通知 |
| **Signals** | 监听事件、触发副作用 | 业务逻辑 |
| **Services** | 提供可复用的业务能力 | 流程控制 |

### ✅ 数据流向单一

**工作空间管理：**
```
initiate_scan (创建) 
    → scan.results_dir (保存到数据库)
    → task_kwargs (传递给子任务)
    → subdomain_discovery (使用)
```

- ✅ 单一数据源（`Scan.results_dir`）
- ✅ 避免多次更新覆盖
- ✅ 职责明确（创建者负责保存）

### ✅ 信号驱动的事件处理

```
Celery Task 事件 → Signal Handler → Service 方法
```

- ✅ 解耦任务逻辑与状态管理
- ✅ 扩展性强（可添加新 handler）
- ✅ 关注点分离

### ✅ 注册表模式的工作流管理

```python
workflow_registry = {
    'subdomain_discovery': (matcher, builder, name),
    # 易于扩展新工作流
}
```

- ✅ 避免 if-else 链
- ✅ 易于添加新工作流
- ✅ 配置与实现分离

### 问题 2：信号重复注册风险（中优先级）

**位置：** `signals/registry.py#23-54`

```python
def register_all_signals():
    # 每次调用都创建新实例
    status_handler = StatusUpdateHandler()
    notification_handler = NotificationHandler()
    cleanup_handler = CleanupHandler()
    
    # 注册信号
    task_prerun.connect(status_handler.on_task_prerun, weak=False)
    # ...
```

**问题：**
- ❌ 多次调用 `register_all_signals()` 会创建多个实例
- ❌ 虽然 `weak=False` 防止垃圾回收，但会导致重复触发
- ❌ 缺少重复注册检测

**影响：**
- 同一个事件可能被处理多次
- 状态可能被重复更新

**建议方案：**

```python
# 使用模块级单例 + 注册标记
_handlers_registered = False
_status_handler = None
_notification_handler = None
_cleanup_handler = None

def register_all_signals():
    global _handlers_registered, _status_handler, _notification_handler, _cleanup_handler
    
    if _handlers_registered:
        logger.warning("信号处理器已注册，跳过重复注册")
        return
    
    # 创建单例
    _status_handler = StatusUpdateHandler()
    _notification_handler = NotificationHandler()
    _cleanup_handler = CleanupHandler()
    
    # 注册信号...
    
    _handlers_registered = True
    logger.info("信号处理器注册完成")
```

### 问题 3：工作流扩展性不足（中优先级）

**位置：** `orchestrators/workflow_registry.py#13-38`

```python
def _build_subdomain_only_workflow(target, task_kwargs, config):
    # 只支持单任务工作流
    workflow = subdomain_discovery_task.si(target=target, **task_kwargs)
    return workflow, ['subdomain_discovery']
```

**问题：**
- ❌ 当前只支持单任务工作流
- ❌ 缺少任务链（chain）支持
- ❌ 缺少任务并行（group）支持
- ❌ 缺少复杂工作流（chord）支持

**影响：**
- 无法实现"子域名发现 → 端口扫描 → 漏洞扫描"的链式流程
- 无法实现多个扫描工具并行执行

**建议方案：**

```python
from celery import chain, group

def _build_full_scan_workflow(target, task_kwargs, config):
    """构建完整扫描工作流：子域名发现 → 端口扫描 → 漏洞扫描"""
    workflow = chain(
        subdomain_discovery_task.si(target=target, **task_kwargs),
        port_scan_task.si(target=target, **task_kwargs),
        vulnerability_scan_task.si(target=target, **task_kwargs)
    )
    return workflow, ['subdomain_discovery', 'port_scan', 'vulnerability_scan']

def _build_parallel_subdomain_workflow(target, task_kwargs, config):
    """构建并行子域名发现工作流"""
    # 多个工具并行执行
    parallel_tasks = group(
        amass_task.si(target=target, **task_kwargs),
        subfinder_task.si(target=target, **task_kwargs),
        assetfinder_task.si(target=target, **task_kwargs)
    )
    return parallel_tasks, ['amass', 'subfinder', 'assetfinder']
```

### 问题 4：命令执行安全风险（中优先级）

**位置：** `services/command_executor.py#46-54`

```python
result = subprocess.run(
    command,
    shell=True,  # ⚠️ 存在命令注入风险
    check=True,
    # ...
)
```

**问题：**
- ❌ 使用 `shell=True` 存在命令注入风险
- ❌ 如果 `target` 参数来自用户输入，可能被恶意利用
- ❌ 缺少输入验证和转义

**影响：**
- 潜在的安全漏洞
- 恶意用户可能执行任意命令

**建议方案：**

```python
import shlex

class ScanCommandExecutor(CommandExecutor):
    def execute_scan_tool(self, tool_name: str, command: str, target: str = None) -> bool:
        """
        执行扫描工具命令（安全版本）
        
        Args:
            tool_name: 工具名称
            command: 命令模板（不含目标参数）
            target: 目标域名（将被安全转义）
        """
        # 验证目标域名格式
        if target and not self._is_valid_target(target):
            raise ValueError(f"Invalid target format: {target}")
        
        # 使用列表形式传递参数，避免 shell 注入
        cmd_parts = shlex.split(command)
        
        # 使用 shell=False
        result = subprocess.run(
            cmd_parts,
            shell=False,  # ✅ 安全
            check=True,
            # ...
        )
    
    @staticmethod
    def _is_valid_target(target: str) -> bool:
        """验证目标域名是否合法"""
        # 域名正则验证
        import re
        pattern = r'^[a-zA-Z0-9]([a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(\.[a-zA-Z0-9]([a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$'
        return bool(re.match(pattern, target))
```

**或者，如果必须使用 shell=True：**

```python
import shlex

def execute(self, command: str, capture_output: bool = False) -> Optional[str]:
    # 对命令参数进行转义
    safe_command = shlex.quote(command)
    
    result = subprocess.run(
        safe_command,
        shell=True,
        # ...
    )
```

### 问题 7：task_ids 和 task_names 管理分散（低优先级）

**位置：** 
- `tasks/initiate_scan.py#125-126`
- `services/scan_status_service.py#152-200`

```python
# initiate_scan: 不更新 task_ids/task_names
# StatusUpdateHandler.on_task_prerun: 通过信号更新
```

**问题：**
- ❓ `initiate_scan` 已知任务信息，但不保存
- ❓ 完全依赖信号在运行时追加
- ❓ 如果信号处理失败，任务信息会丢失

**影响：**
- 数据完整性依赖信号系统
- 调试困难

**建议方案（可选）：**

```python
# 方案 1: initiate_scan 预先保存任务信息（推荐）
def initiate_scan(self, scan_id: int, engine_id: int):
    # 构建工作流
    workflow, task_names = orchestrator.dispatch_workflow(scan, config)
    
    # 预先保存任务名称
    scan.task_names = task_names
    scan.save()
    
    # 执行工作流（task_ids 由信号追加）
    workflow.apply_async(queue='main_scan_queue')

# 方案 2: 保持现状（信号驱动）
# 优点：解耦
# 缺点：依赖信号系统
```

### 问题 8：缺少事务一致性保证（低优先级）

**位置：** `signals/status_update_handler.py#69-112`

```python
def on_task_success(...):
    # 更新 ScanTask 状态
    self.status_service.update_task_status(...)
    
    # 检查扫描完成
    self.status_service.check_scan_completion(scan_id)
```

**问题：**
- ❌ 两个数据库操作不在同一事务中
- ❌ 如果 `check_scan_completion` 失败，`update_task_status` 已经提交
- ❌ 可能导致状态不一致

**影响：**
- 极端情况下可能出现状态不同步

**建议方案：**

```python
from django.db import transaction

def on_task_success(...):
    with transaction.atomic():
        # 更新 ScanTask 状态
        self.status_service.update_task_status(...)
        
        # 检查扫描完成
        self.status_service.check_scan_completion(scan_id)
```

### 问题 9：日志级别不统一（低优先级）

**位置：** 多个文件

```python
# 有些地方用 info
logger.info("任务开始执行 - ...")

# 有些地方用 debug
logger.debug("任务没有 scan_id 参数，跳过状态更新")

# 有些地方用 warning
logger.warning("Engine %s 没有配置，跳过任务调度", engine_id)
```

**问题：**
- ❓ 缺少统一的日志级别规范
- ❓ 关键操作和调试信息混在一起

**建议：**

制定日志级别规范：

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| **DEBUG** | 调试信息、内部状态 | "追加任务记录"、"Validated subdomains" |
| **INFO** | 正常业务流程 | "任务开始执行"、"扫描完成" |
| **WARNING** | 预期内的异常 | "配置为空"、"工具执行失败" |
| **ERROR** | 系统错误 | "数据库操作失败"、"目录创建失败" |
| **CRITICAL** | 严重错误 | "服务不可用" |
