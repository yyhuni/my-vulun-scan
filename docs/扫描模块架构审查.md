## 🔴 严重问题

### 1. **队列配置问题（死锁风险）**

**问题：** 所有任务都在同一个 `scans` 队列
```python
# initiate_scan_task.py
@shared_task(name='initiate_scan', queue='scans', bind=True)

# subdomain_discovery_task.py  
@shared_task(name='subdomain_discovery', queue='scans')
```

**风险：**
- ⚠️ **死锁可能**：父任务调度子任务，但两者竞争同一 worker
- ⚠️ **资源竞争**：编排任务（轻量）和扫描任务（重量）混在一起
- ⚠️ **无法扩展**：无法针对不同任务类型调整并发策略

**建议：**
```python
# 编排任务 - 快速队列
@shared_task(name='initiate_scan', queue='orchestration')

# 扫描任务 - 执行队列
@shared_task(name='subdomain_discovery', queue='scan_execution')
```

### 2. **Service 层职责不清晰**

**问题：** [SubdomainDiscoveryService](cci:2://file:///Users/yangyang/Desktop/scanner/backend/apps/scan/services/subdomain_discovery_service.py:18:0-403:86) 既是业务服务又包含底层实现

```python
# subdomain_discovery_service.py
class SubdomainDiscoveryService:
    def discover(self, target: str, base_dir: str):
        # 创建目录
        # 执行命令
        # 读取文件
        # 合并结果
        # ...
```

**问题：**
- ❌ 违反单一职责原则
- ❌ 难以测试（底层操作和业务逻辑耦合）
- ❌ 难以复用（所有逻辑绑定在一个类中）

**建议架构：**
```python
# 服务层 - 业务编排
class SubdomainDiscoveryService:
    def discover(self, target, base_dir):
        # 编排业务流程
        pass

# 执行器层 - 工具执行
class AmassExecutor:
    def execute(self, target, output_file):
        # 只负责执行 amass
        pass

class SubfinderExecutor:
    def execute(self, target, output_file):
        # 只负责执行 subfinder
        pass

# 工具层 - 文件操作
class ResultMerger:
    def merge(self, file_list):
        # 只负责合并文件
        pass
```

## 🟡 中等问题

### 3. **Signal Handler 中直接查询数据库**

```python
# cleanup_handler.py:66
scan = Scan.objects.get(id=scan_id)  # ❌ Handler 直接查数据库
workspace_dir = scan.results_dir
```

**问题：**
- ❌ Handler 应该只负责信号分发，不应处理数据访问
- ❌ 违反分层原则

**建议：**
```python
# Handler 只调用 Service
def on_task_postrun(self, ...):
    self.cleanup_service.cleanup_task_files(scan_id, task_name)

# Service 处理数据访问
class CleanupService:
    def cleanup_task_files(self, scan_id, task_name):
        scan = Scan.objects.get(id=scan_id)
        # ...
```

### 4. **缺少 Repository 层**

**当前架构：**
```
Controller → Service → Model (直接 ORM)
```

**问题：**
- ❌ Service 层直接操作 ORM，耦合度高
- ❌ 数据访问逻辑散落在多个 Service 中
- ❌ 难以切换数据源（如从 Django ORM 迁移到其他）

**建议引入 Repository：**
```python
# Repository 层
class ScanRepository:
    def get_by_id(self, scan_id: int) -> Scan:
        return Scan.objects.get(id=scan_id)
    
    def update_status(self, scan_id: int, status: str):
        # ...

# Service 使用 Repository
class ScanStatusService:
    def __init__(self):
        self.repo = ScanRepository()
    
    def update_status(self, scan_id, status):
        scan = self.repo.get_by_id(scan_id)
        # ...
```

### 5. **缺少 DTO（数据传输对象）**

**问题：** 直接传递 Model 对象或字典
```python
# 返回字典，结构不明确
return {
    'total': saved_count,
    'target': target,
    'result_file': result_file
}
```

**建议：**
```python
from dataclasses import dataclass

@dataclass
class SubdomainScanResult:
    total: int
    target: str
    result_file: str
    scan_id: int
```

## 🟢 轻微问题

### 6. **错误处理不统一**

```python

except Scan.DoesNotExist:
    ...

# 有的地方捕获所有异常
except Exception as e:  # noqa: BLE001
```

```python
# exceptions.py
class ScanException(Exception):

class ScanNotFoundException(ScanException):

class ScanExecutionException(ScanException):
    pass
```

## 📊 架构层级问题总结

### 当前架构
```
Controller (API)
    ↓
Service (业务逻辑 + 数据访问 + 底层实现)  ← 太臃肿
    ↓
Task (异步执行)
    ↓
Signal Handler (状态更新 + 数据访问)     ← 职责混乱
    ↓
Model (ORM)
```

### 建议架构
```
Controller (API)
    ↓
Service (业务编排)                     ← 只负责编排
    ↓
Repository (数据访问)                   ← 统一数据访问
    ↓
Task (异步执行)
    ↓
Executor (工具执行)                     ← 独立工具层
    ↓
Signal Handler (事件分发)               ← 只负责分发
    ↓
Model (ORM)
```
