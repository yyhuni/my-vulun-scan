## 关键问题（高优先级）
- **[缺失后端接口，与前端调用不一致]**
  - 前端调用：`EndpointService.getEndpointsByDomainId(domainId)` → `GET /api/v1/domains/{id}/endpoints`
    - 后端现状：未在 `backend/routes/endpoint.go` 注册该路由；Swagger 文档中也不存在该路径（`backend/docs/docs.go` 未检出）。
  - 前端调用：`EndpointService.updateEndpoint(data)` → `PUT /api/v1/endpoints/{id}`
    - 后端现状：未定义对应 handler/route。
  - 前端调用：`EndpointService.removeFromOrganization(data)` → `POST /api/v1/endpoints/remove-from-organization`
    - 后端现状：未定义对应 handler/route。
  - 影响：相关功能将返回 404/未路由，导致前端页面操作失败。
  - 建议：补齐路由、handler、service，并完善 Swagger 注释与统一响应类型。

- **[服务层忽略排序参数]**
  - `services.EndpointService.GetEndpoints(req)` 固定 `Order("updated_at desc")`，忽略了 `req.SortBy/req.SortOrder`：`backend/internal/services/endpoint.go`
  - `GetEndpointsBySubdomainID(...)` 同样忽略排序入参。
  - 建议：安全拼接白名单字段（url, method, status_code, created_at, updated_at），按传入的 sortBy/sortOrder 排序。

- **[列表过滤契约不一致]**
  - 前端 `GetEndpointsRequest` 包含 `domainId/subdomainId/method/statusCode`：`front/types/endpoint.types.ts`
  - 后端 `models.GetEndpointsRequest` 仅有 `page/page_size/sort_by/sort_order`：`backend/internal/models/endpoint.go`
  - `handlers.GetEndpoints()` 仅绑定现有 4 个字段，服务层也未实现过滤。
  - 建议：后端扩展请求结构体与查询逻辑，支持前端已定义的过滤字段，确保总数统计与数据查询一致。

- **[批量删除错误处理不健壮]**
  - `handlers.BatchDeleteEndpoints()` 基于 `err.Error()[0:2]` 字符串切片判断错误类型，存在潜在索引越界风险且语义脆弱：`backend/internal/handlers/endpoint.go`
  - 建议：
    - 定义明确的业务错误（sentinel errors）或错误码，并在 handler 中基于错误类型/码分支；
    - 至少使用 `strings.HasPrefix` 并在切片前检查长度。

- **[详情接口未加载关联数据]**
  - `services.EndpointService.GetEndpointByID()` 未 `Preload("Domain").Preload("Subdomain")`，与 `GetEndpoints()` 的行为不一致：`backend/internal/services/endpoint.go`
  - 建议：对详情接口也 `Preload`，与列表保持一致数据体验。

- **[健康检查未使用统一响应结构]**
  - `/health` 返回 `gin.H`：`backend/routes/infrastructure.go`
  - 建议：统一使用 `models.APIResponse` 返回结构化响应（或将该路由标记为非 API，不受统一响应约束）。

- **[Swagger 注释重复]**
  - `handlers.GetEndpointsBySubdomainID()` 的注释块出现重复参数声明：`backend/internal/handlers/endpoint.go`
  - 建议：清理重复，避免生成文档噪音。

## 其他问题（中-低优先级）
- **`response` 包路径与文档不一致**
  - 规则记载 `backend/internal/utils/response.go`；当前为 `backend/internal/response/response.go`。
  - 建议：统一规范文档或考虑迁移（低优先级，不影响功能）。

- **关联清理策略**
  - 当前主要依赖外键 `OnDelete:CASCADE`（模型：`Endpoint.Domain/Subdomain/Vulnerabilities`）：`backend/internal/models/endpoint.go`
  - 规则推荐 `Select(clause.Associations)` 与 `Association` 方法；现有实现可工作，如遇更复杂聚合删除，可按需增强。

- **分页统计可维护性**
  - 未来若加入过滤，请确保 `Count()` 与 `Find()` 使用同一过滤条件，避免总数与数据不一致。

- **前端类型与真实数据结构偏差**
  - `Endpoint` 类型中 `domain/subdomain` 定义为 `string`：`front/types/endpoint.types.ts`
  - 后端返回为对象（当 `Preload` 时），可能与期望不符。虽当前 UI 未使用该字段，但存在一致性风险。
  - 建议：
    - 将其定义为对象类型（如 `Domain | null`、`Subdomain | null`）；或
    - 后端在响应层提供冗余字段 `domain_name/subdomain_name`，前端继续用字符串显示。

## 建议的修复顺序（落地清单）
- **P0｜补齐接口**
  - 新增 `GET /api/v1/domains/{id}/endpoints`（路由、handler、service、Swagger）。
  - 新增 `PUT /api/v1/endpoints/{id}`（更新端点）。
  - 新增 `POST /api/v1/endpoints/remove-from-organization`（解除组织-端点关联，结合 `Association`）。
- **P0｜排序/过滤一致性**
  - `GetEndpoints` 与 `GetEndpointsBySubdomainID` 支持排序参数；扩展 `GetEndpointsRequest` 支持 `domain_id/subdomain_id/method/status_code` 并实现过滤。
- **P1｜错误处理与状态码**
  - 批量删除/单个删除：不存在 ID 时应返回 404 或 400，而非一律 500。避免通过字符串切片判断错误类型。
- **P1｜数据一致性**
  - `GetEndpointByID` 也 `Preload` 关联；前端 `Endpoint` 类型调整或后端补冗余名字段。
- **P2｜统一响应**
  - `/health` 改为统一响应结构，或在项目规范中将其排除为非 API 路由。
- **P2｜文档整洁度**
  - 去重 `GetEndpointsBySubdomainID` 注释项。

- **按子域名列出端点**
```bash

```
- **（缺失）按域名列出端点 - 期望新增**
```bash

```
- **创建端点（自动提取域名/子域名，缺失则跳过）**
```bash
curl -X POST 'http://localhost:8888/api/v1/endpoints/create' \
  -H 'Content-Type: application/json' \
  -d '{
    "endpoints": [
      {"url": "https://org1-example-03.com/logout", "method": "POST", "status_code": 404, "title": "Logout", "content_length": 4096}
    ]
  }'
```
- **删除端点（当前不存在 ID 时返回 500，建议改造为 404/400）**
```bash
curl -X DELETE 'http://localhost:8888/api/v1/endpoints/999999'
```
