## 🔴 严重问题

### 1. 生产环境日志泄露敏感信息

**问题描述**：  
所有 React Query hooks 文件中的 `console.log` 和 `console.error` 没有使用环境变量控制，会在生产环境暴露敏感数据（用户信息、API响应、错误详情等）。

**影响文件**：
- `hooks/use-organizations.ts` (8处)
- `hooks/use-domains.ts` (10处)
- `hooks/use-subdomains.ts` (6处)
- `hooks/use-endpoints.ts` (6处)
- `hooks/use-tools.ts` (2处)

**问题代码示例**：
```typescript
// ❌ 错误：没有环境变量控制
onSuccess: (response) => {
  console.log('创建组织成功')
  console.log('后端响应:', response)  // 暴露完整响应数据
}

onError: (error: any) => {
  console.error('创建组织失败:', error)
  console.error('后端响应:', error?.response?.data || error)  // 暴露错误详情
}
```

**修复方案**：
```typescript
// ✅ 正确：使用环境变量控制
onSuccess: (response) => {
  if (process.env.NODE_ENV === 'development') {
    console.log('创建组织成功')
    console.log('后端响应:', response)
  }
}

onError: (error: any) => {
  if (process.env.NODE_ENV === 'development') {
    console.error('创建组织失败:', error)
    console.error('后端响应:', error?.response?.data || error)
  }
}
```

**修复优先级**: 🔴 最高（安全问题）

## 🟡 中等问题

### 2. API 响应字段命名理解错误（文档问题）

**问题描述**：  
历史记忆中存在错误的修复建议，声称"API 响应使用下划线命名，响应数据保持原始格式"，但实际上 `api-client.ts` 的响应拦截器会自动将下划线转换为驼峰。

**实际情况**：
- 后端返回：`{ status_code: 200, content_length: 1024 }`
- 响应拦截器转换（第280行）：`response.data = camelcaseKeys(response.data, { deep: true })`
- 前端接收：`{ statusCode: 200, contentLength: 1024 }`

**正确使用方式**：
```typescript
// ✅ 正确：前端类型定义和列定义都应该使用驼峰命名
export interface Endpoint {
  statusCode: number | null       // ✅ 驼峰
  contentLength: number | null    // ✅ 驼峰
  createdAt: string               // ✅ 驼峰
  updatedAt: string               // ✅ 驼峰
}

// ✅ 正确：表格列定义也使用驼峰
{
  accessorKey: "statusCode",      // ✅ 驼峰
  accessorKey: "contentLength",   // ✅ 驼峰
  accessorKey: "updatedAt",       // ✅ 驼峰
}
```

**需要修正的文档**：
- 历史记忆中关于"列定义应该使用下划线"的说法是错误的
- 需要澄清：只有**请求参数**（params/data）在拦截器中被转换为下划线，**响应数据**会被转换为驼峰

**修复优先级**: 🟡 中等（文档/理解问题，不影响功能）

### 4. 错误提示策略不一致

**问题描述**：  
在 React Query hooks 中，错误处理方式不统一：
- 有些直接使用后端的 `response.message`
- 有些前端自己构造错误提示"XXX失败，请稍后重试"
- 缺少统一的错误处理策略

**问题代码示例**：
```typescript
// 情况1：使用后端 message
if (response.state !== 'success') {
  throw new Error(response.message || '创建域名失败')
}

// 情况2：前端自己构造
onError: (error: any) => {
  toast.error('创建域名失败，请稍后重试')  // 忽略后端的具体错误信息
}
```

**建议策略**：
1. **成功提示**：前端自己构造（如"成功创建 5 个域名"）
2. **错误提示**：优先使用后端的 `message`，提供更具体的错误原因
3. **控制台日志**：开发环境打印完整的 `error.response.data`

**修复优先级**: 🟡 中等（用户体验）

## 📝 备注

### 关于 API 响应命名转换机制

**完整流程**：
```
1. 后端返回 (snake_case)
   { status_code: 200, created_at: "2024-..." }
   
2. 响应拦截器转换 (api-client.ts:280)
   response.data = camelcaseKeys(response.data, { deep: true })
   
3. 前端接收 (camelCase)
   { statusCode: 200, createdAt: "2024-..." }
   
4. 前端使用
   - 类型定义：interface Endpoint { statusCode: number }
   - 列定义：accessorKey: "statusCode"
   - 组件使用：row.getValue("statusCode")
```

**请求流程**：
```
1. 前端发送 (camelCase)
   { pageSize: 10, sortBy: "created_at" }
   
2. 请求拦截器转换 (api-client.ts:206/217)
   config.data = snakecaseKeys(config.data)
   config.params = snakecaseKeys(config.params)
   
3. 后端接收 (snake_case)
   { page_size: 10, sort_by: "created_at" }
```

**关键点**：
- ✅ 前端代码全程使用驼峰命名
- ✅ 拦截器自动处理请求/响应的命名转换
- ❌ 不要在列定义中使用下划线（这是历史错误理解）
